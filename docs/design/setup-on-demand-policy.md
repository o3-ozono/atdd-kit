# setup-* オンデマンド移管の設計方針

Issue #370。`docs/design/setup-eager-copy-inventory.md` で「参照で足りる」と判定した成果物のうち、
書き込み・秘匿値を伴わないものを対象に、setup-* を実行していない環境でも不足を検出しユーザーへ
プロンプトする統一パターンを定義する。

## 移管対象ごとの定義

各移管対象について「トリガー（どの skill・コマンドが呼ばれたとき）」「検出ロジック（何が不足しているか判定する方法）」
「プロンプト方法（通知 / 自動修復 / confirm 要求）」の 3 要素を定義する。

| 移管対象 | トリガー | 検出ロジック | プロンプト方法 |
|---------|---------|-------------|----------------|
| W-3a: GitHub ラベル不足検出 | `autopilot` / `full-autopilot` の起動節（Iron Law の各ゲート前） | `commands/setup-github.md` の 16 ラベル定義を正準ソースとし、`gh label list` の結果と突き合わせて不足ラベル名を列挙する（`scripts/check-required-labels.sh`） | 不足があればラベル名を通知するのみ（エラー終了しない）。`--create` 指定時のみ `gh label create --force` で不足分を自動修復する |
| W-3b: hook plugin-global 配線 | すべてのセッション起動（`hooks/hooks.json` は plugin ロード時に常時有効） | `hooks/hooks.json` の全 `command` が `${CLAUDE_PLUGIN_ROOT}` 参照であることを回帰 AT で pin（実行時の検出は不要 — setup-* を経由せず最初から plugin-global） | 該当なし（検出・プロンプト不要。setup-* に依存しない不変条件として doc・回帰 AT で保証） |
| （将来拡張の型） sim-pool-guard.sh 等の plugin-global 参照化 | 対応する MCP ツール呼び出し（PreToolUse） | プロジェクトローカルコピーが存在しない場合、plugin 参照パスへフォールバックする分岐を hook 側に実装 | 検出失敗時は通知のみ・処理は継続（後続 Issue でスコープ確定時に本表へ行追加） |

## pre-flight check の標準ガードパターン

すべてのオンデマンド pre-flight check は、以下の標準パターンに従う。

- **エラー終了しない**: 不足を検出してもスクリプト・hook は非ゼロ終了しない。標準出力（または stderr）に不足内容を通知するのみで、呼び出し元のワークフローを止めない。
- **スキップ可能**: 検出そのものが失敗する環境（依存コマンド不在・認証不備など）では「スキップした」旨を通知して正常終了する。ユーザーに追加のセットアップを強制しない。
- **confirm 後続行**: 自動修復を伴う経路（例: ラベル作成）は、既定では通知のみに留め、ユーザーが明示的に confirm した場合（例: `--create` フラグ、または対話的な confirm）にのみ副作用のある操作を実行する。

この 3 原則（エラー終了しない・スキップ可能・confirm 後続行）が、W-3a のラベル不足検出をはじめとする
すべての pre-flight check 実装が満たすべき最小契約である。

## 冪等性チェックリスト

- [ ] **ラベル作成の冪等化**: `gh label create --force` を用いる。既存ラベルへの再作成が副作用ゼロ（色・説明の上書きのみ、エラーにならない）であることを利用し、「既に存在すれば何もしない」を実質的に満たす。2 回連続実行しても既存ラベルは変更されず、エラー・重複作成が起きない。
- [ ] **hook 配線の冪等化**: hook は plugin-global（`${CLAUDE_PLUGIN_ROOT}` 参照）で常時有効なため、setup-* を再実行しても新たなコピーが作られず重複適用が起きない。プロジェクトローカルへのコピーという概念自体が存在しないため、「差分のみ適用」が自動的に成り立つ。
- [ ] **検出スクリプトの再実行安全性**: `scripts/check-required-labels.sh` は読み取り専用の検出（`gh label list`）を既定動作とし、副作用のある操作（`--create`）を明示フラグで分離することで、検出のみの再実行は常に安全（副作用ゼロ）である。

## Open Questions への回答

- **UX を壊さない（Open Question 2）**: 標準ガードパターンの「エラー終了しない」「スキップ可能」がこの要件に直接対応する。
- **冪等性（Open Question 3 / Outcome 3）**: 上記チェックリストがラベル・hook の両経路をカバーする。
