# Job Story

> **Loaded by:** `defining-requirements` (Problem / Outcome 節)

See [common-principles.md](common-principles.md) for the operating principles shared across all 4 techniques in this catalog.

## 目的

ユーザーの行動をペルソナ（〜としては）ではなく状況（〜のとき）を起点に記述することで、「誰であるか」ではなく「どんな状況でどんな動機から行動するか」に焦点を当てる。ペルソナに紐づく先入観を排し、状況・動機・期待結果の因果関係を明確にする。

## 問いの型

- 「それはどんな状況で起きますか？（When _____）」
- 「そのとき、何を達成したいと思っていますか？（I want to _____）」
- 「それが達成されると、どんな結果が得られますか？（So I can _____）」
- persona（「〜として」）を問わず、状況（Situation）→ 動機（Motivation）→ 期待結果（Expected Outcome）の順で深掘りする。

## 適用先マッピング

- `defining-requirements` の **Problem 節**: 「現状の困りごと」を状況起点（When）で言語化する際に使う。
- `defining-requirements` の **Outcome 節**: 「達成したい測定可能な状態」を期待結果（So I can）として言語化する際に使う。

## 一次情報

Alan Klement, "Replacing the User Story with the Job Story", *JTBD (Jobs-to-be-Done) blog*, November 2013.

## 例

> 聞き手: 「その作業に取り掛かるのは、どんな状況のときですか？」
> 回答者: 「複数の Issue が同時に ready 状態になったとき」
> 聞き手: 「そのとき、何を達成したいですか？」
> 回答者: 「人が張り付かなくても、順番に処理が進んでいてほしい」

構造化すると: *When 複数 Issue が同時に ready になったとき、I want to 順番に自動処理が進んでほしい、so I can 人が張り付かずに済む。*
