# UI 設計 doc2 — ビジュアル方針・実装連携の規律

> **Loaded by:** designing-ui skill

`designing-ui` スキルの後半2フェーズ（ビジュアル方針・実装連携）が従う規律を定義する。doc1（`designing-ui-doc1.md`）が確定した骨格（画面単位・階層・遷移・配置）に対して、**装飾の文法**とプラットフォーム別の実装作法を適用し、実装者への handoff まで仕上げる。

---

## ビジュアル方針フェーズの規律

### 装飾の文法

骨格に装飾を与える判断は、以下の観点で行う。

- **ゲシュタルト原則**（近接・類同・連続・閉合）— doc1 のワイヤーで示した要素のグルーピング「意図」を、視覚的にどう表現するか。
- **Typography** — フォントファミリー・サイズスケール・行間の選択。
- **Color** — 配色・コントラスト比（アクセシビリティの WCAG 2.2 達成基準と接続）。
- **Spacing** — 余白のスケール、グリッドシステム。
- **Component 選択** — 既存 Design system のコンポーネントから何を使うか。

### プラットフォーム別作法の選択意思決定

「どう見せるか」は対象プラットフォームの規約に従う。規約を要約・翻訳せず、参照規格を明示し、選択の根拠を記録する。

| プラットフォーム | 参照規約 |
|-----------------|----------|
| iOS / macOS | HIG（Human Interface Guidelines） |
| Android | Material Design |
| Web | Baseline |

規約に根拠がない独自の意思決定は `[独自]` と明示する。読み手が「これは規約由来か、独自判断か」を区別できるようにする。

### Design system 再利用の規律

既存の Design system がある場合、新規コンポーネント・トークンを起こす前に既存資産の再利用を検討する。再利用しない場合はその理由を記録する。

- **成果物。** `docs/issues/<NNN>/visual-policy.md` にプラットフォームと Design system の選択根拠を書く。

## 実装連携フェーズの規律

### アクセシビリティ実装連携

doc1 のワイヤーフェーズから組み込んだアクセシビリティ方針を、実装可能な粒度まで具体化する。

- **WAI-ARIA ロール** — インタラクティブ要素・ランドマークに割り当てるロールを明記する。
- **WCAG 2.2 達成基準** — コントラスト比・フォーカス可視性・タップ領域サイズ等、該当する達成基準を明記する。
- **JIS Z 8520** — 国内向けプロダクトで参照する場合、該当項番と対応方針を注記する。

### Handoff メモの粒度

エンジニアが実装に着手できる粒度で、以下を明記する。

- **コンポーネント** — 使用するコンポーネント名と Design system 上の対応。
- **トークン** — 色・タイポグラフィ・スペーシングのデザイントークン名。
- **アクセシビリティ注記** — 上記 WAI-ARIA / WCAG 2.2 / JIS Z 8520 の適用箇所。

- **成果物。** `docs/issues/<NNN>/implementation-handoff.md` にコンポーネント・トークン・アクセシビリティ注記を書く。
