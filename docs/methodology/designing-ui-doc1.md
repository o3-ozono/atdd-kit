# UI 設計 doc1 — UI 要件・情報設計・ワイヤーフレームの規律

> **Loaded by:** designing-ui skill

`designing-ui` スキルの前半3フェーズ（UI 要件確認・情報設計・ワイヤーフレーム）が従う規律を定義する。この3フェーズは**骨格・配置・画面遷移までを確定する**工程であり、装飾（色・フォント・余白などのビジュアル方針）は一切含めない。装飾とプラットフォーム別実装作法の規律は doc2（`designing-ui-doc2.md`）が担う。

---

## UI 要件フェーズの規律

- **機能要件との対応付けが出発点。** PRD の機能要件を1件ずつ、「どの画面に現れるか」の問いで画面へマッピングする。マッピングされない機能要件を残さない。
- **問いは引き出し型（pull）。** 作り手が既に持っている画面イメージを、こちらが決め打ちせず問いで引き出す。仕様のないところを勝手に埋めない。
- **成果物は対応表。** `docs/issues/<NNN>/ui-requirements.md` に機能要件 ↔ 画面の対応表を書く。

## 情報設計フェーズの規律

- **画面の単位は情報設計フェーズで確定する。** ワイヤーフレームに着手する前に、「これは1画面か、複数画面に分けるか」を固める。ワイヤーの最中に画面単位を変更すると骨格の手戻りが大きい。
- **階層と遷移を先に決める。** 画面同士の親子関係・遷移フロー（どこから来てどこへ行くか）を情報設計の段階で明文化する。
- **成果物は画面リストと遷移図。** `docs/issues/<NNN>/information-architecture.md` に画面単位・階層・遷移を書く。

## ワイヤーフレームフェーズの規律

- **骨格・配置・遷移のみを記述する。** ASCII アートや Mermaid 図等で、要素の配置と画面間の遷移経路を表現する。
- **装飾を含めない。** 色・フォント・余白・アイコンの具体的な見た目はこのフェーズのスコープ外。装飾を混入させると「骨格として正しいか」の判断が装飾の好みに引きずられる。
- **アクセシビリティはこのフェーズから組み込む。** WAI-ARIA のランドマーク・見出し構造・フォーカス順序といった構造レベルのアクセシビリティは、骨格の一部として最初から考慮する（後付けにしない）。
- **成果物は骨格記述。** `docs/issues/<NNN>/wireframes.md` に骨格・配置・遷移を書く。

## ゲシュタルト原則の位置づけ

近接・類同・連続・閉合などのゲシュタルト原則は、**要素同士の視覚的なグルーピングの原則**であり、本質的には装飾判断に属する。doc1（骨格フェーズ）では扱わず、doc2（ビジュアル方針フェーズ）の文脈に位置づける。doc1 のワイヤーフレームでは、グルーピングの「意図」（この要素群は関連している）までを配置で示し、その意図を視覚的にどう表現するか（近接・類同など）は doc2 の責務とする。
