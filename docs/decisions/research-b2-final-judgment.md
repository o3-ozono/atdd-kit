# Research B2 — 総合判断と最終レポート

**作成日:** 2026-04-13
**対象 Issue:** #31 セッション統計収集の価値検討
**担当:** Researcher B2（統合担当）
**入力:** A1（メトリクス・収集機構評価）、A2（プライバシー・保守・役割検出）、B1（PoC 実測）

---

## 1. Executive Summary

セッション統計収集は **段階的導入（Phase 1 のみ即実施）** を推奨する。
Transcript 解析 + Stop フック方式は zero-dependencies 制約に適合し、LLM 推論時間の 0.5% 未満のオーバーヘッドで skill/agent/command の実行状況を取得できる（B1 実測）。
プライバシーリスクは「最小収集の原則」で制御可能であり、実装コストは低い（A2）。
ただし Issue #45（PO 稼働確認）は統計収集なしで静的解析により即時解決できるため、本投資判断から切り離す（A2 §3.4）。

---

## 2. 判断: **段階的導入**（Phase 1 即実施、Phase 2 以降は Go/No-Go 評価後）

「やる / やらない / 段階的導入」の三択において **段階的導入** を選択する。

理由:
- 「やる（フル実装）」は現時点では過剰。OTel やリアルタイム集計など、現在のユースケースに不要な機能を含むリスクがある
- 「やらない」は機会損失。skill/agent 実行状況の可観測性は Issue 駆動ワークフローの品質向上に直結する価値がある
- 「段階的導入」により、Phase 1 の低コスト実装で価値を先に検証してから Phase 2 以降を判断できる

## 3. 根拠（価値 vs コスト）

| 価値項目 | 具体的なメリット | 根拠 |
|---------|--------------|------|
| ワークフロー遵守率の可視化 | skill チェーン完走率（discover→ship）を定量測定できる | A1 §1-A |
| autopilot 利用状況の把握 | agent タイプ別 spawn 数から Developer/QA/Reviewer の実際の稼働状況が判明 | A1 §1-E、A2 §3.2 |

| 実装コストの低さ | bash + python3 のみで実現。atdd-kit の zero-dependencies 制約に完全適合 | A1 §2-2、B1 §1 |
| オーバーヘッドの小ささ | 1 event ≈ 16ms / 94 bytes。200 events のセッションでも LLM 推論時間の ~0.5% | B1 §2-A、§3 |

### 3.2 コスト側の評価

| コスト項目 | 定量評価 | 根拠 |
|-----------|---------|------|

| 実行時オーバーヘッド | ~16 ms/event（macOS）。PostToolUse は非ブロッキングのためユーザー体感への影響は限定的 | B1 §2-A、§4-1 |

| プライバシーリスク | tool_input（コマンド全文・ファイル内容）は HIGH リスク。最小収集の原則で制御可能 | A2 §1.3、§1.5 |
| 継続保守 | 低〜中。Claude Code API 変更時の追随が年 1〜2 回程度想定 | A2 §2.1、§2.4 |

## 4. 段階的導入ロードマップ

### Phase 1: 最小実装（即実施推奨）

**対象メトリクス:** skill 発火回数、agent 生成数（タイプ別）、command 使用頻度、セッション時間、tool 分布

**実装内容:**
1. `hooks/session-stats`（bash スクリプト）を新規作成
2. `hooks/hooks.json` の `Stop` イベントに登録
3. `scripts/analyze-session.py`（python3）を新規作成
4. 収集フィールドは最小セット（A2 §4 推奨）のみ:
   ```json
   {"schema_version": 1, "session_id": "...", "hook_event_name": "...", "timestamp": "...", "tool_name": "...", "agent_type": "..."}
   ```
5. 保存先: `$XDG_CACHE_HOME/atdd-kit/stats/` にローカル JSONL

**プライバシー制御:**
- `tool_input`（コマンド全文・ファイル内容）は収集しない
- `prompt`（ユーザー入力）は収集しない
- `transcript_path` の内容は読まない（Stop フック起動後の事後解析スクリプトが transcript を読むのは acceptable）

**工数見積:** 1〜2 日（スキーマ設計・スクリプト実装・BATS テスト）

**Go 判定基準（Phase 1 完了時）:**
- [ ] skill 発火回数・agent タイプ別 spawn 数が正しく記録されることを実セッションで確認
- [ ] プライバシー懸念フィールドが収集されていないことをログ検査で確認
- [ ] ログサイズが 100MB/月 を超えていない（容量管理上限）

### Phase 2: 分析スクリプト整備（Phase 1 Go 後）

**追加内容:**
1. `scripts/report-session.sh` — 月次サマリを stdout に出力する集計スクリプト
2. ログローテーション（月次圧縮・90 日超分の自動削除）
3. opt-out 機能: `.claude/workflow-config.yml` に `telemetry: false` で収集無効化

**Go 判定基準（Phase 2 完了時）:**
- [ ] 月次レポートが実際のワークフロー改善アクションに繋がった実績が 1 件以上ある
- [ ] スキーマ変更時のマイグレーション手順が `docs/` に文書化されている

### Phase 3: 高度分析（Phase 2 Go 後、必要に応じて）

**追加内容（条件付き）:**
- SubagentStart/Stop フックでのリアルタイム記録（agent 稼働実績の即時可視化）
- GitHub API との相関分析（Issue/PR 進捗 × skill 実行数）

**No-Go 条件（Phase 3 を実施しない場合）:**
- Phase 2 の月次レポートが実際に参照・活用されていない
- 保守負荷（Claude Code API 追随）が予測を超えて増加している

## 5. Issue #45 との切り分け結論

**結論: Issue #45 は本投資判断から切り離す。統計収集は Issue #45 の必要条件ではない。**

A2 §3.4 が示す静的解析で以下が即座に判明する:

| 分析項目 | 静的解析結果 |
|---------|-------------|
| PO エージェントの参照経路 | `commands/autopilot.md` のみ（team-lead として参照）。skills/ からの直接参照なし |
| autopilot 以外での PO 利用 | `agents/README.md` に `claude --agent po` の例示があるが、実セッションでの使用証拠なし |
| skills/ からの呼び出し | `grep -r "po" skills/` で結果なし |

**推奨:** Issue #45 は「autopilot 実行時にのみ PO が team-lead として spawn される設計を明文化し、通常ワークフローでの PO 不使用を仕様として確定する」方向で、静的解析に基づいて解決する。セッション統計収集の完成を待つ必要はない。

## 6. 却下した選択肢と理由

| 選択肢 | 却下理由 |
|--------|---------|
| **OTel フル導入** | zero-dependencies 制約違反。外部 Collector（Prometheus/ClickHouse 等）の構築が必要で個人開発規模では過剰。skill/agent 固有メトリクスは transcript 解析の方が容易（A1 §2-3、§4 見送り推奨） |
| **Hooks のみでリアルタイム収集（全ツール）** | プロセス起動コストが支配的（macOS: ~7.5ms/event）。全ツールに matcher を設定すると大規模セッション（1,000 events）で ~15 秒の累計オーバーヘッド。matcher で絞れば解決するが、それなら Stop フック + Transcript 解析の方がシンプル（B1 §4-2） |
| **UserPromptSubmit フックでの prompt 収集** | プライバシーリスク HIGH。ユーザーが要件・設計案・個人情報を含む可能性のあるプロンプトを収集するのは atdd-kit のユーザー信頼を損なう（A2 §1.3） |
| **いきなりフル実装（Phase 1〜3 一括）** | 現時点で Phase 2/3 の価値は未検証。最小実装（Phase 1）で価値を先に確認してから投資を拡大するのが原則（A2 §4 推奨: 段階的導入） |
| **「やらない」** | skill/agent 実行状況の可観測性はワークフロー品質向上に直結する価値があり、Phase 1 の実装コストは低い（1〜2 日）。機会損失を正当化する根拠が弱い |
