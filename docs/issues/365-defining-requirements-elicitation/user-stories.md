# User Stories: defining-requirements に引き出し型対話ディシプリンを追加

## Functional Story

### F1. Iron Law: 対話ディシプリン ブロックの追加

**I want to** `skills/defining-requirements/SKILL.md` の Flow セクションに独立した `## Iron Law: 対話ディシプリン` ブロックを持ち、そこに 6 つのディシプリン（1 ターン 1 問 / 引き出し型 HARD-GATE / 語彙制約 / 経緯記録の自動駆動 / Wall 検知と差し戻し / ターゲット層化追問）が一括で明記されている,
**so that** `defining-requirements` を実行するたびに対話の質を担保する行動規範が一貫して適用され、後工程（User Story・Plan・AT）が AI 補完の仮説ではなく作者から引き出した本質に立脚できる.

### F2. 提案完成型 HARD-GATE と例外境界の明文化

**I want to** 課題・ゴール・機能の本質について AI が候補を完成形で列挙して作者に選ばせる出力を禁止し、その例外として手法領域（何をどう計測・実装するか）のみ選択肢提示を許す境界線が SKILL.md に明文化されている,
**so that** 本質は作者から引き出され、手法は AI が支援するという役割分担が明確になり、作者の思考が「選ぶだけ」で止まる事態を防げる.

### F3. Wall 検知シグナルと差し戻し文言テンプレート

**I want to** 3 つの Wall シグナル（層化要素の欠如 / 情報量の極端な低下 / 症状の別語再陳述）を検知し、1 回深掘りしてなお情報量が上がらなければ上流の壁打ちへ戻す提案を発動する手順と、共通の差し戻し文言テンプレート（発動シグナルを本文中で補足）が SKILL.md に内蔵されている,
**so that** 作者が思考の行き詰まりに当たったまま情報量の枯渇した回答で PRD が組み上がることを止め、質を回復してから先へ進める.

### F4. ターゲット層化追問

**I want to** 「全員」型の層化されない回答に対し「強いて言うと誰？」を 1 回だけ追問して層化を試みる動作が SKILL.md に規定されている,
**so that** ターゲット・課題の解像度が上がり、層化されたインサイトを PRD に反映できる.

### F5. Iron Law の構造 pin テスト

**I want to** `tests/acceptance/` に新規 BATS ファイルを追加し、Iron Law 6 項目それぞれの識別キーフレーズが SKILL.md 本文に含まれることを `grep` で検証する pin テストが存在する,
**so that** 将来の編集で対話ディシプリンが欠落・退行した場合に自動検出でき、ディシプリンの存在が構造的に保証される.

### F6. CHANGELOG 更新と version bump

**I want to** `CHANGELOG.md` を Keep a Changelog 形式で更新し、`DEVELOPMENT.md` の Versioning 規約に準拠した patch version bump を行う,
**so that** SKILL.md の対話動作層への変更が変更履歴として追跡可能になり、リリース規約が守られる.

## Constraint Story (Non-Functional)

### C1. 対話の語彙制約

**I want to** AI 応答が内部 ID・フレームワーク用語・出典名を用いず、作者の過去回答はその語をそのまま引用し、原則を平易な言葉に言い換えた状態で運用される,
**so that** 作者は専門用語や内部識別子に阻まれることなく自分の言葉で思考でき、引き出し型対話が成立する.

### C2. 経緯記録の PRD 本体への定着

**I want to** 会話中に機能の増減・優先度変更・分類変更を検知したら、変更内容・理由・日付が git commit message ではなく PRD 本体に追記されている,
**so that** 意思決定の経緯が成果物そのものに残り、後から参照する際に commit 履歴を辿らなくても変更理由を追跡できる.
