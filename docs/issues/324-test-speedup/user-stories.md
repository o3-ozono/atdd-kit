# User Stories: atdd-kit 自身のテスト高速化（メタテスト撤去＋影響度ベース並列ランナー）

## Functional Story

**I want to** `tests/acceptance/AT-271.bats` の入れ子フルスイート再実行（AT-006）を撤去し、必要なら影響限定の軽量チェックへ置換できる,
**so that** 当該ファイルが 118s → 約 2s に短縮され、ローカルフルスイート時間がほぼ半減し、開発ループと autopilot の AT ゲートのボトルネックが解消される.

**I want to** GNU parallel 等の外部依存なしで実行時にコア数を検出（`nproc` → `sysctl -n hw.ncpu` → `getconf` → フォールバック4）し、重み均衡ファイルシャーディングで `--all`（全件）と `--impact --base <ref>`（影響範囲のみ）の両モードを並列実行できる `scripts/run-tests.sh` を使える,
**so that** 既存 `bats_runner.sh` / `impact_map.sh` と整合した形で、マシン非依存に全件スイートを従来直列より明確に高速に回せる.

**I want to** 「ユーザー最終レビュー前＝必ず全件 / ATDD 各イテレーション＝修正箇所への影響範囲のみ」というフェーズ別実行ポリシーがフロースキル・ドキュメントに明文化され、claude 系 e2e（`tests/e2e/*.bats` 等）もローカル/CI の置き場所ではなく影響度基準で回す/回さないが判断されるよう統合されている,
**so that** 物理的な置き場所ではなく「何を修正したか」という本質的な基準でテスト実行範囲が最適化され、各フェーズで無駄なく必要十分なテストが回る.

**I want to** 現状ローカル限定/CI 限定で回している live e2e 等のテストの実行条件（skip ガード・専用ワークフロー）が棚卸しされ、影響度基準との対応として整理・文書化されている,
**so that** 既存の実行分けが物理的置き場所ではなく影響度基準で説明できる状態になり、実行可否の判断根拠が一貫する.

**I want to** フェーズ別実行ポリシー（影響度＋フェーズ別）が atdd-kit の配布メソドロジー（`docs/methodology/` 等、適用先プロジェクトに展開されるドキュメント）に明文化されている,
**so that** atdd-kit を適用する各プロジェクトでも「最終レビュー前＝全件 / ATDD 各回＝影響範囲のみ」という同じドクトリンを標準として共有できる.

## Constraint Story (Non-Functional)

**I want to** AT-006 撤去後も #271 の回帰意図が AT-271 の他 AT（構造的検査）と CI のフルスイートで担保され、回帰検出力が落ちていない,
**so that** メタテスト撤去による高速化が、回帰の見逃しという品質低下と引き換えにならない.

**I want to** 並列ランナーが GNU parallel 等の外部依存なしで動作し、コア数を実行時検出することで特定マシン構成に依存しない,
**so that** 開発者ごとの環境差や CI ランナー構成に関わらず、どこでも同じ並列ランナーがそのまま動く.

**I want to** 本 Issue が影響範囲テスト選択ロジック（`impact_map.sh` / `impact_rules.yml`）を新規実装せず既存基盤の活用に限定し、仕組み（impact 選択ツール）の一般化・配布（#323）と重複しない,
**so that** 既に存在し回帰テスト済みの選択ロジックを作り直さず、#323 とのスコープ重複・重複実装を避けて保守コストを増やさない.
