# PRD: atdd-kit 自身のテスト高速化（メタテスト撤去＋影響度ベース並列ランナー）

## Problem

atdd-kit 自身の bats スイート（114 ファイル / 約 1582 テスト）が遅く、開発ループおよび autopilot の AT ゲートのボトルネックになっている。#308 作業中に実測・原因特定した。

**現状:**

- フルスイート直列が遅い。並列化（`bats --jobs 8` / ファイルシャーディング）を試しても 1.6〜1.8x 程度で頭打ちになる。
- 頭打ちの主因は単一の自己再帰メタテスト `tests/acceptance/AT-271.bats` の **AT-006**（行313-336）。これは `run bats "tests/" <全 acceptance>` でスイートほぼ全体を入れ子で丸ごと再実行する（#271 の回帰ガード意図）。
- `bats -r tests/` ではスイートが1回走り、AT-271 がもう1回ネストで走るため全体時間がほぼ倍になる。
- **本 Issue で実測（2026-06-18）:** `AT-006` 単独 = **116.44s**。AT-271.bats 全体 118s の約 98% を占め、ファイル並列・`bats --jobs` のどちらでも破れない下限になっている。
- テストの実行範囲がフェーズに応じて最適化されていない。ATDD の各イテレーションでも全件回しがちで、修正箇所と無関係なテストまで毎回実行される。
- live LLM を呼ぶ claude 系 e2e（`tests/e2e/*.bats` 等）は重く、ローカル/CI の物理的な置き場所で実行可否が決まっており、「何を修正したか」という本質的な基準で回す/回さないが判断されていない。

**それによって何が困るか:** 開発のフィードバックループが長く、autopilot の AT ゲート（毎イテレーション実行）のコストが高い。冗長な入れ子フルスイート再実行は CI のフルスイートと重複しており、ローカルでの価値に対して時間コストが見合わない。

## Why now

- #308 の調査で根本原因（AT-006 の入れ子再実行）と実測値が確定し、撤去の効果（ローカル時間ほぼ半減）が定量的に裏づけられた。今が低コストで効く改善のタイミング。
- autopilot（#318 full-autopilot 含む）が毎イテレーションで AT ゲートを回すため、スイート高速化の効果が運用全体に複利で効く。
- 影響範囲ベースのテスト選択基盤（`scripts/bats_runner.sh` / `scripts/impact_map.sh` / `config/impact_rules.yml`）が既に存在し、これを活用すれば低コストで実現できる。

## Outcome

完了時に達成されている状態:

1. **メタテスト撤去:** `tests/acceptance/AT-271.bats` の入れ子フルスイート再実行（AT-006）が撤去され、当該ファイルが 118s → 約 2s に短縮される。ローカルフルスイート時間がほぼ半減する。#271 の回帰意図は AT-271 の他 AT（構造的検査）と CI のフルスイートで担保され、回帰検出力を落とさない。
2. **影響度ベース並列ランナー:** 依存なし（GNU parallel 不要）でコア数を実行時検出する並列ランナーが提供され、影響範囲選択（`--impact`）と全件（`--all`）の両モードを並列実行できる。フルスイート（全件）は依存なし並列で従来直列より明確に高速になる。
3. **フェーズ別実行ポリシーの確立:** 「ATDD の各イテレーション = 修正箇所への影響範囲のみ」「ユーザー最終レビュー前 = 必ず全件」という、ローカル/CI ではなく **影響度＋実行フェーズ** に基づく実行基準がフロー/ドキュメントに明文化され、claude 系 e2e もその基準に統合される。
3b. **配布メソドロジーへの展開:** 上記のフェーズ別実行ポリシーが atdd-kit の配布メソドロジー（`docs/methodology/` 等、適用先プロジェクトに展開されるドキュメント）に明文化され、atdd-kit を適用する各プロジェクトでも同じドクトリン（最終レビュー前＝全件 / ATDD 各回＝影響範囲のみ）が標準として共有される。
4. **既存実行分けの整理:** 現状ローカル限定/CI 限定で回しているテスト（live e2e 等）の実行条件が整理・文書化され、物理的置き場所ではなく影響度基準で説明できる状態になる。

## What

スコープ内:

1. **AT-271:AT-006 の撤去（提案1・最大効果・依存なし）** — 入れ子フルスイート再実行を撤去する。実測で AT-006 のコスト（116.44s）はほぼ全て「スイート全体の入れ子再実行」であり、#271 の個別回帰は同ファイルの他 AT が構造的に担保し、フルスイート green は CI が担保するため冗長と確認済み。完全削除を基本線とし、必要なら影響限定の軽量チェックへ置換（具体手段は plan/AT 段階で確定）。
2. **依存なし並列ランナー新設（提案2）** `scripts/run-tests.sh` — 実行時にコア数を検出（`nproc` → `sysctl -n hw.ncpu` → `getconf` → フォールバック4、マシン非依存）し、重み均衡ファイルシャーディングで並列実行する。`--all`（全件）と `--impact --base <ref>`（影響範囲のみ）を既存 `bats_runner.sh` / `impact_map.sh` と整合させて両対応する。
3. **フェーズ別実行ポリシーの配線（提案3を影響度ベースに再解釈）** — 「最終レビュー前＝全件 / ATDD 各イテレーション＝影響範囲のみ」をフロースキル・ドキュメントに明文化する。claude 系 e2e（`tests/e2e/*.bats` 等）も、ローカル/CI の置き場所ではなく影響度（修正がスキル本体等に及ぶか）で回す/回さないを判断する基準に統合する。
4. **既存ローカル/CI 実行分けの棚卸し** — 現状の live e2e 等の実行条件（skip ガード・専用ワークフロー）を整理し、影響度基準との対応を文書化する。
5. **フェーズ別実行ポリシーの配布メソドロジー化** — フェーズ別実行ポリシー（影響度＋フェーズ別）を atdd-kit の配布メソドロジー（`docs/methodology/` 等、適用先プロジェクトに展開されるドキュメント）に明文化し、atdd-kit を適用する各プロジェクトでも標準ドクトリンとして共有できるようにする。これは方針ドクトリンの明文化であり、仕組み（impact 選択ツール）の一般化・配布（#323）とは別軸。

## Non-Goals

- **影響範囲テスト選択の仕組み（ツール）の他プロジェクトへの一般化・配布** — #323 が担当。本 Issue が扱うのは方針ドクトリンの明文化（What 5）までで、impact 選択スクリプト/addon の汎用化・配布実装は #323 に委ねる（重複実装を避けるため）。
- **影響範囲選択ロジック（`impact_map.sh` / `impact_rules.yml`）の新規実装** — 既存基盤を活用する。本 Issue は並列化とフェーズ別配線が主で、選択ロジックの作り直しはしない（既に存在し回帰テスト済みのため）。
- **テスト内容そのものの正しさ・カバレッジ向上** — 高速化と実行ポリシーが目的であり、個々のテストのアサーション改善は対象外（スコープ肥大を避けるため）。
- **CI ワークフローのフルスイート実行の撤去** — CI は引き続きフルスイートを回す。AT-006 撤去の回帰担保が CI に依存するため、CI 側のフルスイートは温存する。

## Open Questions

- AT-006 の撤去方式（完全削除 vs 影響限定の軽量チェック置換）の最終確定 — 実測で完全削除が冗長性の観点で妥当と確認済み。最終形は plan/AT 段階で技術判断する（PRD としては「撤去」で確定）。
- `scripts/run-tests.sh` と既存 `scripts/bats_runner.sh` の役割分担（統合 or 並列化レイヤーとして bats_runner を内部利用）— plan 段階で設計する。
- フェーズ別ポリシーを配線するフロースキルの具体箇所（running-atdd-cycle / reviewing-deliverables / merging-and-deploying のどこに何を書くか）— plan 段階で確定する。
