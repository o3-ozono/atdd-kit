# PRD: docs/methodology/a11y-test-taxonomy.md 新設 — a11y テスト手段の3分割と「自動 green ≠ 達成」の明文化

## Problem

atdd-kit の方法論ドキュメント群に、a11y テストの**手段の分け方**が定義されていない。現状では「自動検査（ルールベース）を CI で回して緑になれば a11y 対応完了」という誤った理解をワーカー（人間・AI 問わず）が持つリスクが常在している。

具体的な問題は二層ある。

1. **誤解の温床**: Playwright の公式ドキュメントおよび Deque の分析が示すとおり、自動ツールが検出できるのは WCAG 違反の**一部のみ**であり、大半は手動でしか発見できない。この事実がプロジェクト内に明文化されていないため、自動検査が緑になった段階で手動レビューを省略するリスクがある。

2. **手段の未整理**: 自動静的解析・自動操作検証・手動レビューの 3 手段が同一の「a11y テスト」として扱われており、それぞれがカバーする範囲・粒度・実行タイミングが不明瞭。どの手段で何を検証するかの共通認識がなく、カバレッジの抜けや重複が生じやすい。

## Why now

複数の web / iOS / Android 向けセットアップスキルが追加・整備されつつあり、各スキルが a11y 対応指針を参照する場面が増えている。この段階で土台となる分類ドキュメントが存在しないと、各スキル・各 Issue が独自解釈で a11y を実装し、方法論の分散が固定化する。また、design review を必須ゲートとして明文化するためには、その根拠（「自動では取れない品質が存在する」）を参照できるドキュメントが先に必要である。

## Outcome

以下の状態が、外部から観察可能な形で達成されている。

- `docs/methodology/a11y-test-taxonomy.md` が存在し、以下の4項目が読み取れる。
  1. a11y テスト手段が「自動・静的」「自動・操作」「手動」の3カテゴリに分類され、各カテゴリの対象・粒度・実行タイミングが説明されている
  2. 「自動 green ≠ a11y 達成」であることが、Playwright ドキュメントおよび Deque 分析を一次情報として明示されている
  3. 適用基準として WCAG 2.2 AA が第一候補、JIS X 8341-3:2016 が WCAG 2.0 相当（版差への留意付き）と記載されている
  4. 「テスト手段の分け方（本ドキュメントの主題）」と「WCAG 適合レベルのトリアージ（どの SC をどこまで狙うか）」が別軸であることが明示されている
- `tests/test_a11y_taxonomy_structure.bats`（または同等の BATS ファイル）が存在し、上記ドキュメントの存在と必須セクションの構造を検証している
- CHANGELOG.md が更新され、`.claude-plugin/plugin.json` の patch version が 1 上がっている

## What

- **`docs/methodology/a11y-test-taxonomy.md` 新規作成**: 以下の構成を持つドキュメント。
  - **テスト手段の3分割**: 「自動・静的」「自動・操作」「手動」の定義と、各手段のカバー対象・粒度・実行タイミング（unit/CI ゲート / E2E / design review）。[独自] の整理であることを明示。
  - **「自動 green ≠ a11y 達成」の明文化**: 自動ツールが WCAG 違反の一部のみを検出するという事実（Playwright docs・Deque 分析を一次情報として引用）と、手動 review が design review を必須ゲートとする理由。
  - **適用基準節**: WCAG 2.2 AA を第一候補とする根拠。JIS X 8341-3:2016 は WCAG 2.0 相当であり版差（WCAG 2.0 ↔ 2.2）に留意が必要である旨の注記。
  - **別軸の明示**: テスト手段分類（本ドキュメントの主題）と WCAG SC レベルトリアージ（適合レベルの選択）が独立した設計判断軸であり混同しないことの明記。
- **BATS 構造検証テスト新規追加**: `docs/methodology/a11y-test-taxonomy.md` のファイル存在と必須セクション（3分割・自動 green ≠ 達成・適用基準・別軸）の構造を検証する BATS テストを `tests/` 配下に追加。
- **CHANGELOG.md 更新** + **`.claude-plugin/plugin.json` patch version bump**（DEVELOPMENT.md Versioning 準拠）

## Non-Goals

- **各プラットフォーム（web / iOS / Android）固有の a11y テスト実装手順の策定** — 本 Issue は「手段の分け方」という方法論の土台を整備する。プラットフォームごとの具体的な実装手順（どのライブラリを使うか、どのコマンドを実行するか等）は別 Issue で扱う。
- **WCAG SC レベルトリアージ（どの SC をどこまで狙うか）** — 適用基準として WCAG 2.2 AA を第一候補と記載するにとどめ、個別 SC の優先度付けやトリアージ方針の策定は別 Issue とする。本 Issue の主題と別軸であることのみ明示する。
- **既存スキルへの a11y ゲートの組み込み** — 本 Issue はドキュメントの新設のみ。setup-web / setup-ios 等の各スキルへの参照追加・ゲート組み込みは別 Issue で扱う。
- **自動 a11y 検査ツールの選定・設定方法の規定** — ツール選定（axe-core / Playwright a11y helpers 等）は実装 Issue の責務。
