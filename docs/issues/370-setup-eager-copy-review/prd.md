# PRD: setup-* の eager-copy を「参照優先 + 使う時に不足検出してプロンプト」モデルへ見直す

## Problem

`commands/setup-*.md`（`setup-github` / `setup-ci` / `setup-discord` / `setup-ios` / `setup-web`）は、実行時に scripts / config / templates をプロジェクトへ**前もって一括コピー（eager copy）**する。しかし plugin 本体はインストール時からディスク上に存在するため、**Claude が読むだけの成果物**（概念 doc・戦略文書・example）はプロジェクトへコピーしなくても参照できる。

この eager-copy モデルには以下の問題がある：

1. **初期 setup コストが過剰**：不要なファイルがプロジェクトに積み上がり、初期摩擦が高い。
2. **鮮度乖離リスク**：コピー物は plugin 本体の更新に追随しないため、参照先と実装が乖離しうる。
3. **責務の不明確さ**：「plugin が提供するもの」と「プロジェクトがオーナーするもの」の境界が曖昧で、何を触ってよいか判断しにくい。

## Why now

plugin アーキテクチャが安定し、skill・doc・config の参照経路が整備されたことで、eager-copy を解体する前提条件が揃った。同時に、ラベル不足検出や hook 配線などすでに個別 issue 化されている未解決の摩擦が複数あり、これらは「on-demand 化」という統一方針の下で一括整理できる。さらに autopilot / full-autopilot の使用頻度が上がるにつれ、setup 漏れ（ラベル未設定・hook 未配線）に起因する実行時エラーの再発コストも増大している。

## Outcome

以下がすべて満たされた状態を完了とする。

1. **仕分け一覧が存在する**：setup-* が現在コピーする全成果物について「参照で足りる（plugin-global 参照）」と「プロジェクトローカルに要る（ユーザー管理が必要）」の二分類が完成し、doc として commit されている。
2. **オンデマンド移管の設計方針が doc 化されている**：「いつ・どの skill が・何を検出して・どのようにプロンプトするか」の設計ルールが明文化されており、今後の個別実装の指針として使える。
3. **低リスク実装が動作している**：仕分け一覧で「オンデマンド移管可」と判定した低リスク項目（ラベル不足検出プロンプト、hook 配線の plugin-global 寄せ）が実装され、冪等に動作することが確認されている（再実行で副作用が重複しない）。

## What

### W-1: 仕分け一覧の作成（doc）

- setup-* の全コピー成果物を洗い出し、各アイテムを下記基準で分類する。
  - **参照で足りる**：概念 doc・戦略・example など、plugin インストール済みなら Claude が参照経路を持つもの
  - **プロジェクトローカルに要る**：カスタム設定・ラベル定義・hook スクリプトなど、プロジェクトごとにオーナーが必要なもの
- 成果物：`docs/design/setup-eager-copy-inventory.md`

### W-2: オンデマンド移管の設計方針 doc

- 移管対象ごとに「トリガー（どの skill・コマンドが呼ばれたとき）」「検出ロジック（何が不足しているか判定する方法）」「プロンプト方法（ユーザーへの通知 or 自動修復 or confirm 要求）」を定義する。
- 冪等性の担保方法（再実行時の二重適用防止）を明記する。
- 成果物：`docs/design/setup-on-demand-policy.md`

### W-3: 低リスク項目の実装着手

以下は冪等性が担保しやすく、既存 setup-* を変更せずに追加実装できる対象とする。

- **ラベル不足検出プロンプト**：各ワークフロー skill（autopilot / full-autopilot 等）の起動時に必須 GitHub ラベルの存在を確認し、不足があれば作成を促す（または confirm 後に自動作成する）。
- **hook 配線の plugin-global 寄せ**：プロジェクト個別設定に頼っていた hook を plugin-global hook として提供し、setup-* を実行していない環境でも機能するようにする。

実装は W-1/W-2 で確定した設計方針に従う。

## Non-Goals

- **既存 setup-* コマンドの即時廃止**：本 Issue では廃止の判断・実行はしない。仕分け結果をもとに廃止対象・廃止手順を定義する作業は別 Issue で扱う。
- **破壊的な移管（MCP config、プロジェクト固有スクリプトなど高リスク項目）**：仕分けで「プロジェクトローカルに要る」と判定したものは本 Issue のオンデマンド移管対象外。
- **setup-* の UI / UX 刷新**：コマンド体系の再設計や新 CLI フローの導入はスコープ外。
- **既存プロジェクトへのバックポート自動化**：本 Issue は plugin 側の変更のみ。既存プロジェクトのコピー物クリーンアップは利用者の判断に委ねる。

## Open Questions

1. **eager-copy と on-demand の境界をどう引くか（W-1 の分類基準）**
   → **解決済み（Gate ① 承認）**：「Claude が plugin インストール済み環境で参照経路を持つか」を一次基準とする。参照可能なら eager-copy 不要。参照不可（ユーザー管理の設定値を含む、または書き込みが必要）ならプロジェクトローカルに要る対象とする。W-1 の仕分け一覧でアイテムごとに判定を記録する。

2. **setup 漏れをその場のプロンプトでどう確実に防ぐか**
   → **解決済み（Gate ① 承認）**：W-3 のラベル不足検出のように、各 skill の起動時ガード（pre-flight check）として実装する。検出失敗はエラー終了ではなく「不足を通知 → ユーザーが confirm してから続行」とし、スキップ可能にすることで UX を壊さない。W-2 にガードの標準パターンを定義する。

3. **冪等性の担保方法**
   → **解決済み（Gate ① 承認）**：各実装は「対象が既に存在すれば何もしない（or 差分のみ適用する）」を原則とする。hook 配線は plugin-global hook の性質（インストール済みなら常に有効）を活用して重複適用を回避する。ラベル作成は GitHub API の「既存なら 422 を無視」パターンで冪等化する。W-2 に冪等性チェックリストを含める。
