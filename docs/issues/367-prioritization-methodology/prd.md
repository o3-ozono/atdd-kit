# PRD: docs/methodology/prioritization.md — 機能優先度の方法論 doc を新設

## Problem

`docs/methodology/scrumban.md` はフロー管理（WIP 制限・プル方式・ラベル遷移）を扱うが、PRD の機能要件を**どう優先度付けするか**の専用方法論が存在しない。

現状の問題は 2 層ある。

1. **実践上の誤用**：「効くけど大変だから WANT に下げる」という判断が常態化しており、本質機能が工数理由で後回しにされる。効き（価値）と工数（コスト）は独立した軸であるにもかかわらず、混在して一つの直感判断に圧縮されている。
2. **プロセス上の欠落**：`defining-requirements` スキルで機能要件を列挙・分類する際、AI が使う優先度基準が明文化されていない。各セッションでの優先度判定が属人的・場当たり的になり、一貫性を保てない。

## Why now

`defining-requirements` スキルの機能要件パートで優先度フレームを参照したい需要が既に顕在化している。方法論 doc がないまま運用を続けると、AI によるセッションごとの判断基準がばらつき、機能要件の分類精度が低いまま固定化される。`scrumban.md` が整備された今、隣接する優先度判断の文書を並べて揃えるのが自然なタイミングである。

## Outcome

- `docs/methodology/prioritization.md` が新設され、5 段階・2 軸・anti-pattern・破棄の扱いが明文化されている
- `defining-requirements` スキルが機能要件の優先度判断時に同 doc を参照できる接続が確立されている
- BATS による構造検証ピンが追加されており、doc の骨格（必須セクション）が欠落しても CI で検出できる

## What

### 新設ドキュメント: `docs/methodology/prioritization.md`

#### 位置づけ

MoSCoW（DSDM: Dynamic Systems Development Method）を**一次情報として忠実に扱い**、atdd-kit の文脈向け派生・拡張として整理する。DSDM 原典から逸脱する部分は `[独自]` で明示する。

#### 5 段階

| ラベル | 意味 |
|--------|------|
| MUST | 今回のスコープで必達。なければ出荷しない |
| SHOULD | 重要だが代替手段があり、欠けても出荷可能 |
| WANT | あれば良いが優先度は低い |
| 次回以降 `[独自]` | 効きは認めるが今回の工数に収まらない。明示的に次スコープへ持ち越す |
| 破棄 `[独自]` | 採用しない。理由付きでコア機能テーブル内に残す（後述） |

#### 2 軸判定 `[独自]`

効き（本質課題への解決力）と工数（今回の実装コスト）を**分けて**評価する。

- **軸 A：効き** — 本質課題をどれだけ直接解決するか
- **軸 B：工数** — 今回のスコープで実装に要するコスト

判定は感覚でよい。AI が漏れた視点（未考慮のリスク・依存関係・代替手段）を差し込む設計とする。2 軸を統合した優先度マッピングの目安は以下の通り。

| 効き | 工数 | 推奨ラベル |
|------|------|-----------|
| 高 | 低〜中 | MUST / SHOULD |
| 高 | 高 | 次回以降（効くが今回は入らない）|
| 低 | 任意 | WANT / 破棄 |

#### anti-pattern

- **工数を効きに混ぜる**：「効くけど大変だから WANT」は誤用。効くなら MUST/SHOULD か「次回以降」が正しい判断
- **視点漏れに気づかず置く**：AI レビューのタイミングを設けず、人間の直感だけで分類を確定させる
- **「次回以降」のバケツ化**：何でも入れて放置し、スコープ越境判断の逃げ場にする
- **破棄理由が空・曖昧**：「不要」などの一言のみで、なぜ採用しないかが追跡できない

#### 破棄の扱い

破棄した機能要件は PRD のコア機能テーブルから削除**しない**。`破棄` ラベルと理由を付けてテーブル内に残す。

目的は**ゾンビ復活防止**：別セッション・別担当者が同じ機能を再提案したとき、「なぜ一度破棄したか」の判断根拠をその場で参照できるようにする。

#### `defining-requirements` との接続

`defining-requirements` スキルの機能要件パートは、本 doc の 5 段階・2 軸フレームを参照して優先度分類を行う。参照指示を同スキルの SKILL.md または loaded-docs リストに追記する。

### BATS 構造検証ピン

`tests/` 配下に BATS テストを追加し、以下のセクションが `prioritization.md` に存在することを CI で検証する。

- 5 段階の定義テーブル（MUST/SHOULD/WANT/次回以降/破棄を含む行）
- 2 軸の定義（効き・工数を含む記述）
- anti-pattern セクション
- 破棄の扱いセクション

### CHANGELOG 更新・バージョン bump

DEVELOPMENT.md の Versioning ルールに従い、CHANGELOG.md を更新し patch version をインクリメントする。

## Non-Goals

- **`prioritization.md` 本文以外のドキュメントへの優先度方法論の複製**：`scrumban.md` や `workflow-detail.md` への転記は行わない。参照リンクのみ追加する
- **GitHub Issue ラベルの設計変更**：`scrumban.md` に記載済みの `M/S/C/W priority` ラベルの廃止・変更は本 Issue のスコープ外
- **`defining-requirements` スキルの実装ロジック変更**：接続は参照指示の追記のみ。スキルの判断アルゴリズム変更は別 Issue で扱う
- **定量的な優先度スコアリング機構**：数値スコアやウェイト計算の導入はしない。本 doc は判断フレームの明文化に留める
