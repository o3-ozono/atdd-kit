# User Stories: User gate を選択肢提示（ワンタップ承認）にする

## Functional Story

<!-- 機能要求を persona 抜き Connextra 形式で記述する。1 ストーリー = 1 ユーザーゴール。 -->

### Story 1: ワンタップ承認

**I want to** すべての User gate（要件承認・設計承認・マージ、および `ok`/`ng` の二択しかないゲートを含む）で、第一選択肢として `承認（ok）` が提示された選択肢からワンタップで承認できる,
**so that** Issue ごとに毎回 `ok` を手入力せずにゲートを通過でき、スマホ/タブレットなど手入力が億劫な環境でも応答コストが下がる.

### Story 2: 文脈に応じた差し戻し選択肢

**I want to** 各ゲートの文脈に応じた `ok` 以外の妥当な選択肢が併せて提示される（要件承認ゲートでは「Problem を修正」「Outcome を修正」「スコープを変更」等の PRD セクション単位、設計承認ゲートでは「User Stories を修正」「Plan を修正」「Acceptance Tests を修正」、マージゲートでは「保留（レビュー継続）」）,
**so that** 「どう返せばいいか」が選択肢で明示され、差し戻し時のセクション指定をワンタップで機械的・確実に行える.

### Story 3: 自由記述の常設

**I want to** どのゲートでも「その他（自由記述）」を常に選択できる,
**so that** 選択肢に収まらないフィードバックを現行の自然言語経路でそのまま返せる.

## Constraint Story (Non-Functional)

### Story 4: 承認/差し戻しロジックへの忠実なマッピング

**I want to** 選択結果が現行の承認/差し戻しロジックにそのままマッピングされる（例: 設計ゲートの非 `ok` は全体差し戻し＋セクション単位 finding 化、部分承認は承認ではない、という現行の意味論は不変）,
**so that** 提示方法を選択肢化しても承認/差し戻しの意味が変わらず、既存の収束ロジックがそのまま機能する.

### Story 5: 非対応チャネルへのフォールバック

**I want to** selection UI が使えないチャネル（ヘッドレス/cron 等）では従来の `ok` テキスト入力にフォールバックできる,
**so that** selection UI に依存せず、どのチャネルでもゲート応答が成立する.

### Story 6: ゲート数の不変性（AL-1）

In order to AL-1（ゲート数固定）を不変に保つ, the system must ゲートの増減を行わず、変えるのは提示方法のみに限定する.
**so that** 選択肢提示化によって承認フローのゲート構造が崩れない.
