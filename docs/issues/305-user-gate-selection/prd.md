# PRD: User gate を選択肢提示（ワンタップ承認）にする

## Problem

atdd-kit の **User gate**（要件承認・設計承認・マージの 3 ゲート、および各フロースキル内の承認ゲート）は、現状ユーザーが毎回ターミナルに `ok` を**手入力**して承認する運用になっている。

- **現状**: ゲート提示は散文のみで、ユーザーは `ok`（承認）や差し戻し内容を**自由記述で手入力**する必要がある。
- **困ること**:
  - ゲートが Issue ごとに複数回（要件→設計→マージ）あり、毎回 `ok` を打つのが手間。
  - スマホ/タブレットなど手入力が億劫な環境では負担が大きい。
  - 承認以外の選択（差し戻し・セクション単位修正・保留など）も自然言語で打つ必要があり、ワンタップで返せず、差し戻し時のセクション指定が機械的・確実でない。
  - `merge してよいか？` のような **`ok`/`ng` の二択しかないゲート**でも、選択肢が提示されずテキスト入力を強いられる。

## Why now

autopilot のドッグフーディングが継続中（#277 以降フル実走運用）で、1 Issue あたり最低 3 回の User gate 応答が**毎回**発生している。ゲート応答コストはセッション数に比例して累積する恒常的な摩擦であり、選択肢提示化はその摩擦を構造的に下げる。Dialog economy（#254）で「最小質問」を整えた今、応答 UX の改善が次の自然なステップ。

## Outcome

完了時、すべての User gate が**選択肢提示（AskUserQuestion 相当の selection UI）**で提示され、ユーザーがワンタップで回答できる。

- すべての User gate（要件承認・設計承認・マージ、および二択しかないゲートを含む）で、第一選択肢に `承認（ok）` が提示される。
- 各ゲートの文脈に応じた `ok` 以外の妥当な選択肢が併せて提示される（差し戻し・セクション単位修正・保留など）。
- 「その他（自由記述）」が常に選択でき、現行の自然言語フィードバック経路が維持される。
- 選択結果が現行の承認/差し戻しロジックに**そのままマッピング**される（例: 設計ゲートの非 `ok` は全体差し戻し＋セクション単位 finding 化）。
- selection UI が使えないチャネル（ヘッドレス/cron 等）では従来の `ok` テキスト入力に**フォールバック**できる。
- AL-1（ゲート数固定）は不変 — 変えるのは**提示方法**のみ。

## What

- 各 User gate のゲート提示を、散文＋テキスト入力から **選択肢提示（AskUserQuestion 相当）** に変更する。
  - **要件承認ゲート**（`defining-requirements`）: 「承認」「Problem を修正」「Outcome を修正」「スコープを変更」… PRD セクション単位の差し戻し選択肢。
  - **設計承認ゲート**（`autopilot`）: 「承認」「User Stories を修正」「Plan を修正」「Acceptance Tests を修正」。
  - **マージゲート**（`merging-and-deploying`）: 「マージ」「保留（レビュー継続）」。
- `ok`/`ng` の二択しかないゲートでも選択肢を提示する（提案者の追加要件）。
- 第一選択肢を常に `承認（ok）` とし、「その他（自由記述）」を常設する。
- 選択結果を現行の承認/差し戻しロジックにマッピングする（差し戻しは現行どおりセクション単位 finding 化）。
- selection UI 非対応チャネルでは従来の `ok` テキスト入力にフォールバックする旨を明記する。

## Non-Goals

- **ゲートの増減**は行わない（AL-1 不変。変えるのは提示方法のみ）。
- **承認/差し戻しの意味論（ロジック）の変更**は行わない（例: 部分承認は承認ではない、非 `ok` は全体差し戻し、という現行ルールはそのまま）。
- **selection UI そのものの実装**は行わない（AskUserQuestion は harness が提供するツールであり、本 Issue はスキル側のゲート提示をそれに合わせて記述・運用するのみ）。
- Dialog economy（#254）の「最小質問」方針の変更は行わない（選択肢提示は最小質問と整合させる）。

## Open Questions

- none remain（提案者が args で「二択ゲートも選択肢提示にする」を明示済み。selection UI 非対応チャネルのフォールバックも Issue 本文で確定済み）。
